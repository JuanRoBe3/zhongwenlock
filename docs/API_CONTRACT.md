# API Contract - ZhongwenLock

## Purpose

This document defines the first version of the API contract for ZhongwenLock.

The API connects:

- the web/PWA app;
- API Gateway;
- Lambda functions;
- DynamoDB;
- the target MVP Custom GPT assisted/direct import flow.

The same ingestion contract can be used during engineering validation with manual JSON samples.

Manual JSON import is not the intended target MVP user experience. It is an engineering validation path used before automating the Custom GPT import flow.

---

## Design Principle

The structured study session output generated by ChatGPT represents what happened during a study session.

ChatGPT can provide:

- detected errors;
- affected concepts;
- suggested categories;
- suggested review priority;
- explanations;
- generated flashcards;
- generated exercises;
- generated mini-tests.

ChatGPT should not provide:

- mastery scores;
- accumulated progress;
- review state;
- ledger balance;
- final penalty amounts.

Those values are calculated by ZhongwenLock.

---

## Import Sources

The backend should track where a study session import came from.

Possible `import_source` values:

```text
manual_validation
custom_gpt_assisted
custom_gpt_direct
```

The import source is used for traceability.

Regardless of the import source, ZhongwenLock must validate the payload and transform it into internal learning items before storing long-term state.

---

## POST /sessions

Imports a completed study session.

This is the most important endpoint for the first product loop.

The endpoint can support:

- manual validation imports during development;
- target MVP imports from the Custom GPT assisted/direct flow.

### Input

The request body follows the example in:

```text
samples/session-event-example.json
```

The payload includes:

- event type;
- schema version;
- user ID;
- import source;
- session metadata;
- score;
- detected errors;
- affected concepts;
- suggested review priority;
- generated flashcards;
- generated exercises;
- generated mini-tests.

### Backend responsibilities

The backend should:

- validate the payload;
- validate product rules;
- reject invalid payloads with clear errors;
- store the imported session trace;
- store detected errors;
- create or update affected concepts;
- store generated flashcards, exercises and mini-tests;
- transform the external event into internal learning items;
- prepare review items using internal rules;
- update review state.

Importing a ChatGPT session does not create a ledger penalty by itself.

The ledger is updated later when the user fails review items inside ZhongwenLock.

### Success response

```json
{
  "status": "ok",
  "message": "Study session processed successfully",
  "session_id": "session_2026_07_05_001"
}
```

### Validation error response

```json
{
  "status": "error",
  "message": "Invalid study session payload",
  "errors": [
    "Missing field: session.id",
    "Invalid review_priority in errors[0]"
  ]
}
```

## GET /review/resume

Returns the next learning block for the user.

This is not only a daily review. The goal is to let the user continue from the last pending learning state.

The backend may return:

- pending flashcards;
- exercises from weak concepts;
- mini-tests;
- items that were previously failed;
- items that should be reviewed again.

---

## POST /review-sessions

Stores the result of a review session completed inside the app.

The app sends all answers together at the end of the review block.

The backend should:

- store the review result;
- store answers;
- update mastery scores;
- update the next review state;
- calculate simulated penalties for failed review answers;
- update the simulated ledger.

Ledger penalties are created here, not during study session import.

---

## GET /dashboard

Returns the global dashboard data.

The dashboard should show:

- global progress;
- estimated HSK level;
- weakest concepts;
- most frequent errors;
- most expensive concepts;
- pending review items;
- simulated ledger total.

---

## GET /concepts

Returns the concept library.

Concepts should be filterable by:

- HSK level;
- category;
- concept type;
- mastery score;
- number of errors;
- accumulated simulated cost.

---

## GET /concepts/{concept_id}

Returns the detail page for one concept.

The detail should include:

- concept information;
- mastery score;
- historical errors;
- corrections;
- related flashcards;
- related exercises;
- related mini-tests;
- accumulated simulated cost.

---

## GET /ledger

Returns the simulated financial ledger.

The MVP does not move real money.

The ledger should show:

- total simulated balance;
- cost by concept;
- cost by category;
- recent entries.

Ledger entries are mainly created when the user fails review items inside ZhongwenLock.

---

## GET /settings/penalty-config

Returns the user's simulated penalty configuration.

Example:

```json
{
  "failed_flashcard_eur": 0.1,
  "failed_exercise_eur": 0.25,
  "failed_mini_test_eur": 0.5,
  "repeated_concept_extra_eur": 0.25,
  "max_per_review_session_eur": 3.0
}
```

---

## PUT /settings/penalty-config

Updates the user's simulated penalty configuration.

This allows the user to decide how much each failed review item costs.

---

## Open Questions

The exact response format for some endpoints is still pending.

For the target MVP, the most important endpoint is:

```text
POST /sessions
```

The rest of the endpoints describe the intended product direction and will be refined as the app is implemented.
